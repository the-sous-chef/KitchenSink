## Goal

Build a **Bidirectional Requirements Traceability Matrix (RTM)** that provides a complete, auditable trail between every requirement, its test cases, and its executable scenarios.

The RTM is the single most scrutinized artifact during any quality or compliance review. Its purpose is to prove — deterministically — that (a) every requirement has been verified by at least one appropriate V&V method (per IEEE 1012), and (b) every test traces back to a justified requirement. No orphaned tests, no untested requirements.

This is a **read-only analysis** command — it generates a report but does NOT modify `requirements.md` or `acceptance-plan.md`.

## The 4 Pillars of Auditable Traceability

Every RTM generated by this command MUST satisfy these 4 pillars:

### Pillar 1: Strict Bidirectionality (The Golden Rule)

Auditors require **bidirectional traceability** — not just forward, but also backward:

- **Forward Traceability (REQ → ATP → SCN):** Proves every requirement has been tested. *Ensures no missing features.*
- **Backward Traceability (SCN → ATP → REQ):** Proves every test maps back to a justified requirement. *Ensures no gold-plating, undocumented features, or rogue code (a massive security/safety risk).*

Both directions are checked and reported.

### Pillar 2: Orphan & Gap Analysis

The RTM doesn't just show what IS connected — it explicitly highlights what is BROKEN:

- **Gaps:** A `REQ` with no `ATP`. The system is incomplete — untested functionality.
- **Orphans:** An `ATP` or `SCN` with no `REQ`. The system has undocumented behavior — a compliance failure.

The script actively scans for and flags both gaps and orphans in a dedicated section.

### Pillar 3: Versioning and Baselines

When requirements change, linked tests become "suspect" until re-verified. The matrix includes:

- **Artifact hashes or timestamps** proving the matrix reflects the CURRENT requirements and test plans, not a stale draft.
- **Source file references** with modification dates for audit trails.

(Deep git-diff-based version tracking is a Phase 2 feature; for the MVP, timestamps and file modification dates establish the baseline.)

### Pillar 4: Granular Execution State

A matrix that only says "this test exists" is useless. Each entry must show its verification status:

- `⬜ Pending Execution` — Test case exists but has not been run
- `✅ Passed` — Test executed and passed (future: linked to CI/CD results)
- `❌ Failed` — Test executed and failed
- `🚫 Blocked` — Test cannot be executed due to a dependency
- `⏸️ Deferred` — Test intentionally deferred with justification

For the MVP, all items default to `⬜ Pending Execution`. Future phases will pull status from CI/CD pipelines or test runners.

### 3. Build the Matrix (Deterministic Script)

Run the matrix builder script — this is a **deterministic operation**, NOT an AI task:

```bash
{SCRIPTS_DIR}/build-matrix.sh {VMODEL_DIR} --output {VMODEL_DIR}/traceability-matrix.md
```
```powershell
{SCRIPTS_DIR}/build-matrix.ps1 {VMODEL_DIR} -Output {VMODEL_DIR}/traceability-matrix.md
```

Where `{SCRIPTS_DIR}` is the parent directory of the `{SCRIPT}` path.

The script:
1. Parses `requirements.md` for all REQ IDs and their descriptions
2. Parses `acceptance-plan.md` for all ATP IDs, descriptions, and linked SCN IDs
3. Cross-references forward (REQ → ATP → SCN) and backward (SCN → ATP → REQ)
4. Identifies gaps (uncovered REQs) and orphans (unlinked ATPs/SCNs)
5. **If `system-design.md` and `system-test.md` exist**, generates Matrix B (Verification) alongside Matrix A (Validation) — parsing SYS components from the Decomposition View and STP/STS test cases
6. **If `architecture-design.md` and `integration-test.md` exist**, generates Matrix C (Integration Verification) — parsing ARCH modules from the Logical View and ITP/ITS test cases
7. **If `module-design.md` exists (with `architecture-design.md`)**, generates Matrix D (Implementation Verification) — parsing MOD modules and their parent ARCH mappings; if `unit-test.md` also exists, includes UTP/UTS test cases
8. **If `hazard-analysis.md` exists (with `system-design.md`)**, generates Matrix H (Hazard Traceability) — parsing HAZ entries, their mitigations (REQ/SYS), and resolving to verification test cases (ATP/STP)
9. Generates the complete traceability matrix with coverage metrics
10. Writes output to `{VMODEL_DIR}/traceability-matrix.md`

**Five-matrix output (v0.5.0+):**
- **Matrix A — Validation (User View):** REQ → ATP → SCN. Always present.
- **Matrix B — Verification (Architectural View):** REQ → SYS → STP → STS. Only present when system-level artifacts exist.
- **Matrix C — Integration Verification (Module Boundary View):** SYS → ARCH → ITP → ITS. Only present when architecture-level artifacts exist. Includes parent REQ annotations in SYS column and CROSS-CUTTING pseudo-rows.
- **Matrix D — Implementation Verification (Module View):** ARCH → MOD → UTP → UTS. Only present when module-design.md and architecture-design.md exist. Each ARCH cell includes parent SYS identifiers in parentheses; `[CROSS-CUTTING]` modules display `([CROSS-CUTTING])` instead of SYS lineage; `[EXTERNAL]` modules display bypass annotations. If unit-test.md is absent, UTP/UTS columns show "⏳ Pending".
- **Matrix H — Hazard Traceability:** HAZ → Mitigation (REQ/SYS) → Verification (ATP/STP). Only present when hazard-analysis.md and system-design.md exist. Gaps where a mitigation has no test coverage show "⚠️ No test coverage".
- Backward compatible: v0.1.0 feature directories (no system artifacts) produce Matrix A only; v0.2.0 produce A + B; v0.3.0 produce A + B + C; v0.4.0 produce A + B + C + D; projects without hazard-analysis.md omit Matrix H.

### 4. Present Results (3 Sections)

Read the generated `traceability-matrix.md` and present the results to the user in the mandatory 3-section format. Your role is to **summarize and interpret** the deterministic output, not to recalculate it.

#### Section 1: The Coverage Audit (The Math)

Auditors look at this first. All numbers are calculated by the script, not the AI.

```
══════════════════════════════════════════════
  TRACEABILITY MATRIX — COVERAGE AUDIT
══════════════════════════════════════════════

  Total Requirements:                  {N}
  Requirements with Test Coverage:     {N} ({pct}%)
  Total Test Cases (ATP):              {N}
  Test Cases with Scenarios:           {N} ({pct}%)
  Total Executable Scenarios (SCN):    {N}

──────────────────────────────────────────────
  FORWARD TRACEABILITY (REQ → ATP → SCN)
──────────────────────────────────────────────
  Untested Requirements:               {N}  {✅ Pass / ❌ FAIL}
  ATPs Without Scenarios:              {N}  {✅ Pass / ❌ FAIL}

──────────────────────────────────────────────
  BACKWARD TRACEABILITY (SCN → ATP → REQ)
──────────────────────────────────────────────
  Orphaned Test Cases:                 {N}  {✅ Pass / ❌ FAIL}
  Orphaned Scenarios:                  {N}  {✅ Pass / ❌ FAIL}

══════════════════════════════════════════════
  OVERALL STATUS: {✅ COMPLIANT / ❌ NON-COMPLIANT}
══════════════════════════════════════════════
```

**Lifecycle-aware metrics:** Coverage counts and percentages are calculated on **active items only** — deprecated items (`[DEPRECATED]`) are excluded from the denominator. Suspect items (`[SUSPECT]`) remain in the active count until resolved. The matrix is **NON-COMPLIANT** if any unresolved suspect items exist, regardless of coverage percentage.

**If a domain overlay is loaded**, present the compliance status framed against the applicable domain standard's traceability requirements (e.g., ASIL-dependent traceability depth for ISO 26262, DAL-dependent objectives for DO-178C). Otherwise, use generic best-practice framing.

### 6. Recommend Next Actions

Based on the analysis:

- **If fully compliant (0 gaps, 0 orphans)**:
  "✅ All requirements are bidirectionally traceable to test cases and scenarios. The RTM is ready for audit review. Proceed with `/speckit.plan` to start implementation planning."

- **If gaps exist (untested requirements)**:
  "❌ Forward traceability failures found. Run `/speckit.v-model.acceptance` to generate missing test cases and scenarios for the {N} uncovered requirements listed in the Exception Report. Then re-run `/speckit.v-model.trace` to rebuild the matrix."

- **If orphans exist (unlinked tests)**:
  "⚠️ Backward traceability failures found. {N} test cases or scenarios do not trace to any requirement. Review the orphans listed in the Exception Report — they may indicate deleted requirements. Remove or reassign orphaned items in `acceptance-plan.md`, then re-run `/speckit.v-model.trace`."

- **If both gaps and orphans exist**:
  "❌ Bidirectional traceability failures found in both directions. Address gaps first (run `/speckit.v-model.acceptance`), then clean up orphans manually, then re-run `/speckit.v-model.trace`."

- **If Matrix D shows ARCH→MOD gaps**:
  "⚠️ Architecture modules without module designs found. Run `/speckit.v-model.module-design` to generate module designs for the {N} uncovered ARCH modules. Then re-run `/speckit.v-model.trace`."

- **If Matrix D shows MOD→UTP gaps (unit-test.md exists)**:
  "⚠️ Module designs without unit test cases found. Run `/speckit.v-model.unit-test` to generate unit tests for the {N} uncovered modules. Then re-run `/speckit.v-model.trace`."

- **If Matrix D shows UTP pending (unit-test.md absent)**:
  "ℹ️ Module designs found but unit test plan not yet generated. Run `/speckit.v-model.unit-test` to generate unit tests, then re-run `/speckit.v-model.trace` to complete Matrix D."

- **If suspect items exist**:
  "⚠️ {N} suspect items found requiring lifecycle review. Suspect items trace to a parent that was deprecated or modified. Run the corresponding V-Model command for each affected layer to resolve suspects (re-parent, deprecate, or confirm active). Then re-run `/speckit.v-model.trace` to verify resolution."

- **If deprecated chains are incomplete**:
  "⚠️ {N} deprecated items have downstream artifacts that are not yet deprecated or re-parented. Review the Deprecated Chains in the Exception Report and run the downstream commands to cascade the deprecation. Then re-run `/speckit.v-model.trace`."

## Operating Constraints

### Read-Only Command

This command MUST NOT modify `requirements.md`, `acceptance-plan.md`, or any other V-Model artifact. It only writes to `traceability-matrix.md`. If issues are found, it recommends corrective actions but does not execute them.

### Deterministic Output

The matrix is built by a script using regex pattern matching, NOT by the AI. The AI's role is limited to:
1. Running the script
2. Reading and presenting the script's output
3. Interpreting gaps and recommending next actions

This ensures the matrix is 100% accurate and audit-ready — no hallucinated coverage claims.

## Governing Standards

This command is governed by the following standards for traceability analysis:

| Standard | Scope | How Used |
|---|---|---|
| **IEEE 1012:2016** | Software Verification and Validation | Traceability analysis verifies each requirement has been verified by an appropriate V&V method (test, analysis, inspection, or demonstration) |
| **ISO/IEC/IEEE 15289:2019** | Content of Life-Cycle Information Items | Content requirements for traceability records: bidirectional linkage, completeness criteria, change history |
| **IEEE 29148:2018** | Requirements Engineering | Requirements traceability as a fundamental property of well-formed specifications |

> **Note:** If a domain overlay is loaded (Step 2), additional regulatory references from the applicable standard (e.g., ISO 26262 Part 6 Clause 9, DO-178C §6.3.4, IEC 62304 §5.7) are presented alongside these best-practice standards.
