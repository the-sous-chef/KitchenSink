# V-Model Requirements Specification: IdP User Authentication

**Feature Branch**: `002-user-auth`
**Created**: 2026-05-09
**Status**: Draft (amended 2026-05-13 — added REQ-045–REQ-050)
**Source**: `specs/002-user-auth/spec.md`

## Overview

IdP-based user authentication for the Commise app across web (Next.js) and mobile (Expo) platforms. Users authenticate via the identity provider (IdP) using Authorization Code Flow with PKCE. Platform-specific flows apply: mobile shows the auth screen automatically; web redirects to the IdP login page. Sessions persist via refresh tokens. Users are identified by auto-generated UUIDv4 IDs synced to the Commise database on signup via a user.created post-registration IdP handler. The feature covers sign-up, login, logout, profile view, account editing, account deletion (cascading to the IdP), password reset, MFA enrollment, social account linking, user impersonation, API authorization, and user suspension/reactivation.

| REQ-019 | The system SHALL provide an account edit page where users can modify their display name, avatar.                                                                                                                                                                                                                                                                                                        | P2       | Users must be able to personalize their identity within the app.                                                                                   | Test                |
| REQ-020 | Changes to display name, avatar on the account edit page SHALL be persisted to the Commise database.                                                                                                                                                                                                                                                                                                  | P2       | Account edits must be durable and reflected across all sessions.                                                                                   | Test                |
| REQ-021 | The account edit page SHALL display the email field as read-only with a note directing users to the IdP for email changes.                                                                                                                                                                                                                                                                                | P2       | Email is managed by the IdP; the app must not allow direct email editing to avoid inconsistency.                                                     | Inspection          |
| REQ-022 | The account edit page SHALL prevent saving when the display name is empty, SHALL display a validation error.                                                                                                                                                                                                                                                                                            | P2       | Display name is a required field; empty values must be rejected client-side.                                                                       | Test                |
| REQ-023 | The system SHALL provide an account deletion flow that requires explicit confirmation (e.g., typing "DELETE") before proceeding.                                                                                                                                                                                                                                                                        | P2       | Account deletion is irreversible; explicit confirmation prevents accidental deletion.                                                              | Test                |
| REQ-024 | On confirmed account deletion, the system SHALL remove the User records plus Account records from the Commise database, then remove the user from the IdP.                                                                                                                                                                                                                                              | P2       | GDPR and trust requirement — all user data must be removed from both systems on deletion.                                                          | Test                |
| REQ-025 | Account deletion SHALL cascade to all user-owned data (recipes, meal plans, etc.) in the Commise database.                                                                                                                                                                                                                                                                                            | P2       | Orphaned user-owned data must not remain after account deletion.                                                                                   | Test                |

| REQ-028 | The password reset flow SHALL be handled entirely by the IdP — the Commise backend SHALL NOT store, process, / have access to user passwords.                                                                                                                                                                                                                                                           | P1       | Security requirement — passwords must never pass through or be stored by the Commise backend.                                                    | Inspection          |
| REQ-029 | The system SHALL provide access to the IdP's MFA enrollment flow from the account settings page.                                                                                                                                                                                                                                                                                                          | P3       | MFA is a security enhancement for users who opt in; access must be surfaced from account settings.                                                 | Test                |
| REQ-030 | The system SHALL support at minimum TOTP (authenticator app) as an MFA method via the IdP.                                                                                                                                                                                                                                                                                                                | P3       | TOTP is the most widely supported and secure MFA method for consumer apps.                                                                         | Test                |
| REQ-031 | When MFA is enrolled, the IdP SHALL require the second factor on all subsequent login attempts.                                                                                                                                                                                                                                                                                                           | P3       | MFA enrollment must be enforced by the IdP on every login to provide the security benefit.                                                           | Test                |

| REQ-033 | The system SHALL allow a logged-in user to unlink a social provider account from their IdP identity, provided at least one login method remains active.                                                                                                                                                                                                                                               | P2       | Users must be able to remove social providers, but must retain at least one login method.                                                          | Test                |
| REQ-034 | When a user links / unlinks a social provider, the system SHALL NOT create a new User / Account record in the Commise database — the existing canonical user ID SHALL be preserved.                                                                                                                                                                                                                   | P2       | Social linking/unlinking is an IdP-level operation; the Commise user identity must remain stable.                                              | Test                |
| REQ-035 | The system SHALL support user impersonation for authorized support/engineering personnel via the IdP's impersonation / token exchange mechanism.                                                                                                                                                                                                                                                          | P2       | Support and engineering personnel need to impersonate users to diagnose issues.                                                                    | Test                |
| REQ-036 | Impersonation sessions SHALL be clearly distinguishable from normal sessions — all API requests made during impersonation SHALL include an impersonation flag, the impersonator's identity in audit logs.                                                                                                                                                                                               | P2       | Audit trail requirement — impersonation actions must be attributable to the impersonator.                                                          | Test                |
| REQ-037 | Impersonation SHALL NOT allow password changes, account deletion, / MFA modifications on behalf of the impersonated user.                                                                                                                                                                                                                                                                               | P2       | Impersonation must be scoped to read/support operations; destructive actions are prohibited.                                                       | Test                |
| REQ-038 | All Commise API endpoints SHALL require a valid IdP access token. Requests without a valid token SHALL receive `401 Unauthorized`.                                                                                                                                                                                                                                                                  | P1       | All API endpoints must be protected; unauthenticated requests must be rejected.                                                                    | Test                |
| REQ-039 | The API Gateway authorizer SHALL validate IdP JWT tokens (signature, expiration, audience, issuer) on every request.                                                                                                                                                                                                                                                                                  | P1       | JWT validation is the security gate for all API access; all claims must be verified.                                                               | Test                |
| REQ-040 | The access token SHALL include the Commise user ID (from `app_metadata`) as a custom claim, enabling the API to identify the requesting user without an additional database lookup.                                                                                                                                                                                                                   | P1       | Performance and simplicity — the user ID in the token avoids a database round-trip on every request.                                               | Test                |
| REQ-041 | The system SHALL support suspending a user by both blocking the user in the IdP (via the Management API "block" flag), setting the User entity `status` to `suspended` in the Commise database.                                                                                                                                                                                                         | P2       | Suspension must be enforced at both the IdP level (blocks new tokens) and the database level (blocks existing tokens).                           | Test                |
| REQ-042 | The API Gateway authorizer SHALL deny access (return `403 Forbidden`) to users whose IdP account is blocked, even if the access token is otherwise valid.                                                                                                                                                                                                                                             | P2       | Suspended users must be denied access even with a currently valid token.                                                                           | Test                |
| REQ-043 | A suspended user who attempts to log in SHALL be shown a clear message indicating their account has been suspended.                                                                                                                                                                                                                                                                                     | P2       | Users must receive actionable feedback when their account is suspended.                                                                            | Test                |
| REQ-044 | The system SHALL support reactivating a suspended user by unblocking them in the IdP, setting the User entity `status` back to `active`.                                                                                                                                                                                                                                                                  | P2       | Suspension must be reversible to support account reinstatement.                                                                                    | Test                |
| REQ-045 | The system SHALL exclusively use Authorization Code Flow with PKCE for all OAuth 2.0 authentication. | REQ-046 | The system SHALL document and communicate to operators that IdP refresh token revocation (on logout or account deletion) is eventually consistent — a revoked token may remain usable for up to 60 seconds. The API Gateway authorizer cache TTL SHALL be set to a maximum of 30 seconds to bound the window during which a revoked token can still authorize requests.                               | P1       | IdP revocation propagation is not instantaneous; the authorizer cache TTL must be constrained to limit the exposure window after revocation.     | Inspection          |
| REQ-047 | When a user is suspended (IdP block applied), the system SHALL acknowledge that existing valid access tokens remain usable until they expire (up to the access token TTL). The access token TTL SHALL be set to a maximum of 300 seconds (5 minutes) to bound the window during which a suspended user can continue to make API requests.                                                             | P2       | Blocking a user in the IdP does not invalidate in-flight access tokens; a short TTL limits the exposure window after suspension.                     | Inspection          |
| REQ-048 | The IdP `post-user-registration` handler fires only for database (email/password) signups and does NOT fire for social provider signups. For social provider signups, the system SHALL use an IdP `post-login` handler to detect first-time social logins (by checking for the absence of `app_metadata.userId`) and perform the same User and Account creation flow as the user.created post-registration IdP handler. | P1       | Without a post-login fallback, social-provider signups will not create Commise database records, leaving orphaned IdP users.                   | Test                |
| REQ-049 | The system SHALL support step-up authentication for sensitive operations (account deletion, MFA enrollment/unenrollment, social account unlinking). Step-up SHALL be implemented via the IdP's `acr_values` / `max_age` parameters to require re-authentication or MFA challenge before the sensitive operation proceeds.                                                                                 | P2       | Step-up authentication ensures that sensitive destructive operations cannot be performed by an attacker who has hijacked an active session.        | Test                |
| REQ-050 | The system SHALL configure an IdP Log Stream to forward IdP tenant logs to CloudWatch Logs. The Log Stream health SHALL be monitored — if the stream is auto-disabled by the IdP (due to repeated delivery failures), a CloudWatch alarm SHALL fire and a Sentry alert SHALL be raised within 15 minutes.                                                                                             | P2       | The IdP may auto-disable log streams after repeated failures; undetected stream disablement creates a silent observability gap in the auth audit trail. | Test                |

## Assumptions

- IdP tenant is already provisioned and configured with the appropriate application registrations (web and mobile) before implementation begins.
- The Commise database (RDS PostgreSQL 16) is available and the schema for `User` and `Account` entities is defined (or will be defined as part of this feature).
- IdP post-registration handlers are available in the IdP tenant plan.
- The `@kitchensink` shared configuration packages (`@kitchensink/typescript`, `@kitchensink/eslint`, `@kitchensink/prettier`, `@kitchensink/vitest`) are published and available in the monorepo.
- Social login with Google is the minimum required provider; additional providers may be added without requiring a spec change.
- The IdP Backend API credentials required for account deletion, social linking/unlinking, suspension, and impersonation will be securely stored in AWS Secrets Manager.
- The reconciliation mechanism will run as a scheduled Lambda function (e.g., EventBridge Scheduler) rather than a real-time process.

## Dependencies

- **IdP tenant** — provisioned with web and mobile application registrations, user.created post-registration handlers enabled, and Management API access.
- **RDS PostgreSQL 16** — the Commise database must be accessible from Lambda functions in the same VPC.
- **AWS API Gateway** — the Lambda authorizer is attached to the API Gateway that fronts all Commise API endpoints.
- **AWS SQS** — required for the version-archive queue referenced in the tech stack.
- **AWS CDK v2** — infrastructure definition and deployment.
- **`@kitchensink` shared configs** — monorepo shared tooling packages must be available.
- **001-commise-recipe-app** — downstream consumer of the authentication layer (FR-045).
- **003-usda-food-data** — downstream consumer of the shared API Gateway authorizer (FR-035).
- **005-ai-integration** — downstream consumer of the auth layer for external agent OAuth (FR-018).
- **010-subscriptions** — downstream consumer of the Account entity's `subscriptionTier` field.

## Glossary

| Term                              | Definition                                                                                                                     |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| IdP (Identity Provider)           | The third-party identity provider used for authentication and authorization.                                                   |
| Authorization Code Flow with PKCE | OAuth 2.0 flow for public clients (mobile/SPA) that uses a code verifier/challenge to prevent authorization code interception. |
| Access Token                      | A short-lived JWT issued by the IdP that grants access to Commise API endpoints.                                               |
| Refresh Token                     | A long-lived token used to obtain new access tokens without user re-authentication.                                            |
| `app_metadata`                    | IdP user metadata that is not modifiable by the user; used to store the canonical Commise user ID.                         |
| UUIDv4                            | Universally Unique Identifier version 4 (random); the canonical user identifier in Commise.                                  |
| Post-Registration Handler         | An IdP handler that fires after a user successfully registers, used to sync the user to the Commise database.               |
| API Gateway Authorizer            | An AWS Lambda function that validates IdP JWTs and returns an IAM policy to API Gateway.                                     |
| Reconciliation                    | The process of detecting and repairing IdP users that lack corresponding Commise database records.                         |
| Impersonation                     | The ability for authorized personnel to act as a specific user for support/debugging purposes.                                 |
| Suspension                        | The state where a user is blocked in the IdP and denied API access; reversible by reactivation.                                  |
| TOTP                              | Time-based One-Time Password; the primary MFA method supported via the IdP.                                                      |
| PKCE                              | Proof Key for Code Exchange; a security extension to OAuth 2.0 for public clients.                                             |
| JWKS                              | JSON Web Key Set; the IdP endpoint that publishes public keys for JWT signature verification.                                |
| Canonical User ID                 | The auto-generated UUIDv4 that is the primary identifier for a user across all Commise systems.                              |
