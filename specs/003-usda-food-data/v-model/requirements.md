# V-Model Requirements Specification: USDA Food Data Integration

**Feature Branch**: `003-usda-food-data`
**Created**: 2026-05-09
**Status**: Draft
**Source**: `specs/003-usda-food-data/spec.md`

## Overview

Integrate USDA FoodData Central as the primary food/nutrition database backing Commise recipes. The system uses an event-driven, queue-based architecture (SQS + Lambda + token bucket) for rate-limited async data fetching. Food data is served from a local PostgreSQL store (with optional Redis cache) — the USDA API is never called in the request path. Cache misses trigger async backfill via EventBridge → SQS → Consumer Lambda, rate-limited to 1,000 USDA API calls per hour.

## Assumptions

- **A-001**: The USDA FoodData Central API rate limit of 1,000 requests per hour per API key is a hard constraint that cannot be increased through paid tiers or support requests within the project timeline.
- **A-002**: The lean launch variant (no Redis, `db.t4g.micro` PostgreSQL) is the default starting configuration. Redis is added when performance thresholds warrant it (p95 read latency > 100ms sustained, or lookup volume > 50K/day).
- **A-003**: Eventual consistency is acceptable for food data. Users tolerate a 10–60 second delay for first-time food lookups in exchange for never blocking on an external API call.
- **A-004**: The USDA API remains publicly available with a free tier and the current `POST /v1/foods` batch endpoint supporting up to 20 IDs per request.
- **A-005**: This feature deploys as an AWS-hosted backend service (Lambda, SQS, EventBridge, RDS, optional ElastiCache) in `us-east-1`. It serves both the web and mobile Commise clients through API Gateway.
- **A-006**: The food data workspace is a new package within the KitchenSink monorepo, following all workspace governance rules from Constitution Principle V.
- **A-007**: Client-side polling (not WebSocket) is the launch notification mechanism. WebSocket (US-9) is deferred until UX testing validates the need.
- **A-008**: The `foods` table schema is purpose-built for this feature. Integration with Commise's `ingredients` entity is a downstream concern handled by feature `001-commise-recipe-app`.
- **A-009**: The USDA API key is stored in AWS Secrets Manager and rotated per AWS best practices. The key is never exposed in client-facing responses or logged. All food data API endpoints share the Commise application's authentication boundary (API Gateway authorizer); no separate auth mechanism is required for this feature.
- **A-010**: All food data API endpoints use URL prefix versioning (`/v1/foods/*`). Breaking changes require a new version prefix (`/v2/foods/*`). The USDA FoodData Central API's own `/v1/` prefix is independent and unrelated to our versioning.

## Glossary

| Term              | Definition                                                                                                                                          |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| `fdcId`           | USDA FoodData Central unique numeric identifier for a food item.                                                                                    |
| Food              | A USDA nutritional record stored locally. Distinct from a Commise Ingredient (a recipe component that MAY link to a Food via `fdcId`).            |
| Ingredient        | A recipe component in the Commise app. MAY link to a Food via `fdcId`; the link is optional.                                                      |
| FetchRequest      | An intent to retrieve food data from the USDA API; manifests as an EventBridge event and SQS message.                                               |
| TokenBucketState  | The rate limiter's current state: `tokens` (float, 0–1,000) and `lastRefill` (ISO 8601). Stored atomically in Redis or PostgreSQL.                  |
| QueueMessage      | An SQS message in the food fetch pipeline containing `fdcId`/`batchIds`, `requestedAt`, `requestSource`, `priority`, and SQS metadata.              |
| FoodDataEvent     | An EventBridge event in the food data lifecycle: `FoodRequested`, `FoodBatchRequested`, `IngestionScheduled`, `FoodDataReceived`, or `FetchFailed`. |
| Tombstone         | A food record with `fetch_status = 'not_found'`, indicating the `fdcId` does not exist in USDA. No further fetch attempts are made.                 |
| Lean Launch       | The default deployment variant: no Redis, PostgreSQL `db.t4g.micro`, token bucket state in PostgreSQL.                                              |
| Full Architecture | The scaled deployment variant: Redis for caching and token bucket, PostgreSQL `db.t4g.small`.                                                       |
| DLQ               | Dead Letter Queue — receives SQS messages that have failed processing 3 times. Retained for 14 days.                                                |
| Token Bucket      | Rate-limiting mechanism with capacity 1,000 tokens, refilling at 16.67 tokens/minute (1,000/hour).                                                  |
| Pending-Fetch     | Deduplication mechanism tracking `fdcIds` currently queued for USDA fetch (Redis Set or PostgreSQL `fetch_status = 'pending'`).                     |
| Stale             | A food record with `fetched_at` older than the configurable staleness threshold (default: 30 days). Eligible for background re-fetch.               |
