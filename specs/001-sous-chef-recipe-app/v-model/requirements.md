**Status**: Draft
**Source**: `specs/001-sous-chef-recipe-app/spec.md`

## Overview

Foundational recipe management for the Sous Chef product: authenticated CRUD on
recipes (title, description, ingredients backed by real food data, ordered
instructions, prep/cook/total time, servings, tags, photos), search and
filtering, soft-delete with GDPR-compliant hard purge, public/private visibility
gated by subscription tier, sharing/cloning of public recipes with attribution
preservation, full version history (last 10 in DB, all versions archived to
S3 with DLQ-backed retry), optimistic concurrency conflict resolution,
collections with snapshot-time cloning and an opt-in "Pull updates" reconcile,
and feature-parity delivery across web and mobile. All other Sous Chef specs
depend on this one.

## Atomization Convention

Per peer review (PRF-REQ-001..029), compound requirements have been decomposed
into atomic sub-requirements identified by lowercase letter suffixes (e.g.,
`REQ-010a`, `REQ-010b`, `REQ-010c`). Each sub-ID is independently testable.
Downstream artifacts (acceptance-plan.md, system-test.md, hazard-analysis.md,
traceability-matrix.md) reference either the parent ID (when verifying the
aggregate behavior) or specific sub-IDs (when verifying a single obligation).

## Assumptions

- Users have internet connectivity for all core features in scope (offline-first behavior is not a requirement of this feature).
- The real food/nutrition database backing REQ-003b/REQ-031 is supplied by feature `003-usda-food-data` (USDA FoodData Central or licensable equivalent).
- Authentication, identity, and subscription-tier resolution (REQ-022, REQ-023, REQ-027b, REQ-CN-004a, REQ-CN-004b, REQ-IF-005a..c) are supplied by feature `002-user-auth`.
- The mobile application targets iOS and Android (REQ-IF-004b).
- "Substantive edit" tracking (REQ-028a, REQ-028b) is durable per recipe-clone and survives subsequent edits (i.e., once unlocked for privacy via a substantive edit, the clone does not need to be re-unlocked on every edit).
- "Authenticated user" in every REQ-NNN above means a user whose Auth0-issued session/JWT is valid and whose subscription tier is known to the API.
- CloudWatch (REQ-NF-016, REQ-NF-017, REQ-041) is the operational monitoring substrate; the corresponding metric/alarm names are defined in `plan.md`.
- The CloudWatch→SNS→PagerDuty integration referenced in REQ-041 is an existing platform service provided by infrastructure outside this feature.
- The product analytics pipeline referenced in REQ-NF-004 (`analytics.events` table, `weekly_core_feature_adoption_pct` metric) is provided by a separate analytics-instrumentation feature; this requirement assumes its existence and remains an Analysis-verified target rather than a system test.

| Optimistic concurrency `version`      | A monotonically increasing integer per recipe used for stale-write detection per REQ-042a/b.                                                                                 |
| Substantive-edit unlock               | The state, per REQ-027b, in which a clone of a public-source imported recipe becomes eligible for private visibility (premium tier + at least one substantive edit applied). |
