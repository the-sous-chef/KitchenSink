### PRF-ACC-001 — Atomic REQ sub-IDs lack ATP linkage despite policy mandate

| Field              | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Severity**       | Critical                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Location**       | Traceability Policy (lines 27–32); ATP-002-A, ATP-005-A/B, ATP-017-A/B, ATP-028-\*, ATP-032-\*, ATP-041-\*, ATP-048-\*, ATP-056-\*                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Description**    | The Traceability Policy (line 30) declares: "ATP linkage uses explicit atomic sub-IDs where requirements were decomposed for independent testability." However, 17 atomic sub-IDs defined in `requirements.md` never appear in any ATP `Linked Requirement` field: `REQ-002a`, `REQ-002b`, `REQ-005a`, `REQ-005b`, `REQ-005c`, `REQ-017a`, `REQ-017b`, `REQ-028a`, `REQ-028b`, `REQ-032a`, `REQ-032b`, `REQ-041a`, `REQ-041b`, `REQ-048a`, `REQ-048b`, `REQ-056a`, `REQ-056b`. Their corresponding ATPs (e.g., ATP-002-A linked to umbrella `REQ-002`) violate the stated policy and produce a false 100% coverage claim because the Coverage Summary counts umbrella IDs, not atomic sub-IDs. The Peer-Review Remediation Log (PRF-ACC-002 entry) claims this was fixed but covers only REQ-035/038/049, not the eight other decomposed requirements. |
| **Recommendation** | Update each affected ATP's `Linked Requirement` field to enumerate its atomic sub-IDs (e.g., ATP-002-A → `REQ-002a, REQ-002b`; ATP-005-A → `REQ-005a, REQ-005b`; ATP-005-B → `REQ-005c`). Where a single ATP cannot validate multiple atomic sub-IDs (e.g., create vs update for REQ-002a/b), split the ATP into atomic-aligned cases following the precedent set by ATP-035, ATP-038, ATP-049. Then re-run `validate-requirement-coverage.sh` and update the Coverage Summary denominator/numerator to reflect atomic-ID accounting.                                                                                                                                                                                                                                                                                                                  |

### PRF-ACC-002 — Coverage Summary baseline contradicts requirements.md

| Field              | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Severity**       | Major                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Location**       | Test Plan Context Exit Criteria (line 25); Traceability Policy (line 29); Coverage Summary (line 1975)                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Description**    | The plan declares the baseline as **"129 requirements"** in three places (exit criteria, traceability policy, coverage summary). However, `requirements.md` defines **137 unique REQ IDs** (counting atomic sub-IDs as the policy requires). The denominator equals the numerator (129/129), which is suspicious — the actual count of distinct REQ IDs _referenced_ in the plan equals 129 only because umbrella IDs are used in place of their atomic decompositions (see PRF-ACC-001).                                                        |
| **Recommendation** | Reconcile the baseline to a single source of truth and reference it from all three locations. Either (a) raise the baseline to 137, recognising atomic sub-IDs as the unit of coverage per policy line 30, and add the missing ATP links per PRF-ACC-001; or (b) document explicitly in the Traceability Policy which atomic sub-IDs are intentionally covered by an umbrella ATP and why splitting was rejected. Run `validate-requirement-coverage.sh` after reconciliation to confirm the script-derived baseline matches the documented one. |

### PRF-ACC-003 — Umbrella REQ IDs referenced where atomic IDs are required

| Field              | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Severity**       | Major                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| **Location**       | ATP `Linked Requirement` fields referencing `REQ-002`, `REQ-005`, `REQ-017`, `REQ-028`, `REQ-032`, `REQ-041`, `REQ-048`, `REQ-056`                                                                                                                                                                                                                                                                                                                            |
| **Description**    | Eight umbrella REQ IDs are referenced in ATP linkage but are not defined as discrete requirement rows in `requirements.md` — they exist only as parent labels for atomic sub-IDs. The Traceability Policy (line 31) tolerates umbrella IDs "for readability" only when atomic links are also present; in these eight cases, no atomic linkage is provided alongside the umbrella, breaking the audit chain.                                                   |
| **Recommendation** | For each affected ATP, replace the umbrella reference with the explicit list of atomic sub-IDs it validates (e.g., `REQ-005` → `REQ-005a, REQ-005b, REQ-005c`). If the umbrella reference is intentionally retained for readability, append the atomic list (e.g., `REQ-005 (REQ-005a, REQ-005b, REQ-005c)`). This finding is a corollary of PRF-ACC-001 and resolves the same way; once atomic IDs appear in linkage, both findings become script-checkable. |

### PRF-ACC-004 — Coverage Summary "100%" claim is misleading under atomic-ID accounting

| Field              | Value                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Severity**       | Major                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Location**       | Coverage Summary (lines 1973–1980); Coverage note (line 1982)                                                                                                                                                                                                                                                                                                                                                                             |
| **Description**    | The Coverage Summary table claims "Requirements with ≥1 ATP — 129 / 129 (100%)". Under the atomic-ID rule mandated by the Traceability Policy, the true coverage is 120 / 137 ≈ 87.6% (17 atomic sub-IDs from PRF-ACC-001 are uncovered). The 100% claim is the most consequential statement in the document and is currently misleading to auditors and downstream consumers (e.g., `validate-requirement-coverage.sh`, `audit-report`). |
| **Recommendation** | After fixing PRF-ACC-001 and PRF-ACC-002, regenerate the Coverage Summary table from script output rather than transcribing it. Add a footnote referencing the validator script and the commit SHA of the requirements.md baseline at the time of validation. Treat any future divergence between the manual table and script output as a CI-blocking error.                                                                              |

### PRF-ACC-005 — Verification Method maintained as document-level enumeration, drift risk

| Field              | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Severity**       | Minor                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Location**       | Verification Method Notes (lines 43–48)                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Description**    | The Inspection/Analysis ATP list is hand-maintained as inline enumerations (22 IDs in the Inspection bullet). When a future ATP is added or its verification method changes in `requirements.md`, this section will silently drift. ISO/IEC/IEEE 29119-3 expects verification method to be a per-test-case attribute, not a document-level enumeration that can desynchronise from the per-case truth.                                                            |
| **Recommendation** | Add a `**Verification Method:** Test \| Inspection \| Analysis` field to each ATP's metadata block (alongside `Linked Requirement`, `Description`, `Validation Condition`, `Expected Result`). Generate the document-level summary list from those per-ATP fields via a script, or remove the summary entirely and rely on the per-ATP field. This brings the plan into structural alignment with `requirements.md`'s per-requirement Verification Method column. |

### PRF-ACC-008 — Peer-Review Remediation Log violates stateless-findings principle

| Field              | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Severity**       | Observation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Location**       | Peer-Review Remediation Log (lines 1988–2003)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Description**    | The remediation log entries reference "source: PRF-ATP-001" through "PRF-ATP-014", but the V-Model peer-review template defines finding IDs as `PRF-{ARTIFACT}-NNN` where `{ARTIFACT}` matches the artifact name (i.e., `PRF-ACC-NNN` for acceptance-plan). The "PRF-ATP-NNN" prefix does not match the template convention. More fundamentally, the template explicitly states findings are stateless and disappear on re-run when fixed; embedding their history in the artifact creates noise and can confuse later reviewers (this current PRF-ACC-001 is a different finding than the historical one cited in the log). |
| **Recommendation** | Remove the "Peer-Review Remediation Log" section. Capture remediation history in commit messages or a separate `peer-review-history.md` file outside the artifact. If the team wants in-artifact provenance, rename the section "Peer-Review Provenance" and constrain it to one line per fix referencing the commit SHA and finding ID, without re-stating the description.                                                                                                                                                                                                                                                 |
