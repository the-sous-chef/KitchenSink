# System Design: Sous Chef - Recipe Management Core

**Feature Branch**: `001-sous-chef-recipe-app`
**Created**: 2026-04-30
**Status**: Draft
**Source**: `specs/001-sous-chef-recipe-app/v-model/requirements.md`

## Overview

This design decomposes the Sous Chef Recipe Management Core into bounded system components aligned to the implemented architecture: NestJS API modules, web/mobile clients, PostgreSQL persistence, S3/CloudFront media storage, and SQS-backed asynchronous version archival. Decomposition follows requirement ownership boundaries (recipe commands, visibility policy, ingredients, search, photos, versioning, collections, GDPR erasure, cross-cutting NFR governance) while preserving many-to-many traceability from every REQ-NNN to one or more SYS-NNN components.

## ID Schema

- **System Component**: `SYS-NNN` — sequential identifier for each component
- **Parent Requirements**: Comma-separated `REQ-NNN` list per component (many-to-many)
- Example: `SYS-003` with Parent Requirements `REQ-001, REQ-005` — component satisfies both requirements

| Source  | Target  | Relationship          | Failure Impact                                                                                                                                                               |
| ------- | ------- | --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SYS-016 | SYS-001 | Calls                 | Web users cannot authenticate/authorize, blocking all protected actions.                                                                                                     |
| SYS-017 | SYS-001 | Calls                 | Mobile users cannot authenticate/authorize, blocking all protected actions.                                                                                                  |
| SYS-016 | SYS-005 | Calls                 | Recipe search/filter unavailable in web UX.                                                                                                                                  |
| SYS-017 | SYS-005 | Calls                 | Recipe search/filter unavailable in mobile UX.                                                                                                                               |
| SYS-016 | SYS-002 | Calls                 | Recipe create/update/delete/clone commands are unavailable in web UX.                                                                                                        |
| SYS-017 | SYS-002 | Calls                 | Recipe create/update/delete/clone commands are unavailable in mobile UX.                                                                                                     |
| SYS-002 | SYS-003 | Uses                  | If SYS-003 is unavailable, SYS-002 cannot validate visibility/substantive-edit rules, causing recipe mutations to be accepted or rejected without proper policy enforcement. |
| SYS-002 | SYS-004 | Calls                 | Ingredient linkage and nutrition enrichment fail during recipe save.                                                                                                         |
| SYS-002 | SYS-006 | Calls                 | Photo attach/confirm flows fail and per-file retry UX degrades.                                                                                                              |
| SYS-002 | SYS-008 | Calls                 | Saves cannot produce required versions or concurrency guards.                                                                                                                |
| SYS-002 | SYS-014 | Writes                | Recipe metadata cannot persist atomically.                                                                                                                                   |
| SYS-005 | SYS-001 | Calls                 | Search endpoint cannot enforce authentication, exposing recipes to unauthenticated callers.                                                                                  |
| SYS-005 | SYS-014 | Reads                 | Search/filter requests time out or return empty/incorrect results.                                                                                                           |
| SYS-006 | SYS-015 | Writes                | Presigned upload and object verification fail, blocking photo ingestion.                                                                                                     |

| SYS-010 | SYS-009 | Subscribes            | Pending archive backlog accumulates because queue work is not drained.                                                                                                       |
| SYS-010 | SYS-014 | Writes                | Pending rows and archive state cannot transition correctly.                                                                                                                  |

| SYS-012 | SYS-014 | Writes                | Cloned collection snapshots and pull reconciliation results cannot persist; clone/pull operations fail without side effects.                                                 |
| SYS-011 | SYS-012 | Calls                 | Clone/pull semantics diverge from snapshot and reconciliation rules.                                                                                                         |
| SYS-013 | SYS-014 | Calls                 | GDPR erase cannot complete DB-side hard purge safely.                                                                                                                        |
| SYS-013 | SYS-015 | Calls                 | S3 versions/photos remain after erasure request completion.                                                                                                                  |
| SYS-013 | SYS-001 | Calls                 | Erasure endpoint cannot verify caller identity, enabling unauthorized erasure requests.                                                                                      |
| SYS-019 | SYS-010 | Reads                 | Archive failures and backlog breaches go undetected operationally.                                                                                                           |
| SYS-020 | SYS-016 | Uses                  | Web quality/accessibility/test conformance drifts from required standards.                                                                                                   |
| SYS-020 | SYS-017 | Uses                  | Mobile quality/accessibility/test conformance drifts from required standards.                                                                                                |

## Operational States

The system exposes the following operational states across its asynchronous lifecycles. These states are referenced by hazard analysis (`HAZ-NNN`) entries and bound to the components that own each state machine. `NORMAL` denotes steady-state synchronous request/response handling outside any of the lifecycles below; `ALL` is reserved for hazards whose severity is invariant across every state.

| STATE               | Owning Component(s)        | Description                                                                                                               | Entry / Exit                                                                                                                                                          |
| ------------------- | -------------------------- | ------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NORMAL              | All synchronous services   | Steady-state online operation handling synchronous API/web/mobile traffic outside any async lifecycle below.              | Default state. Exits transiently when a request enters a specific async lifecycle below.                                                                              |
| PHOTO_PENDING       | SYS-006 (Photo upload)     | Client has obtained a presigned URL and is uploading a photo object to S3 staging (`uploads/...`); no DB row yet.         | Enter: presigned URL issued. Exit: upload completes (→ `PHOTO_PROCESSING`) or aborts/expires (→ `PHOTO_FAILED`).                                                      |
| PHOTO_PROCESSING    | SYS-007 (Photo Lambda)     | Sharp Lambda is normalizing/validating the staged photo and writing the canonical object plus DB linkage row.             | Enter: S3 event triggers Lambda. Exit: success → `NORMAL`; failure → `PHOTO_FAILED`.                                                                                  |
| PHOTO_FAILED        | SYS-006 (Photo upload)     | Photo upload or processing did not complete; staged object may be orphaned and the recipe has no canonical photo linked.  | Enter: upload abort, presigned URL expiry, validation failure, or Lambda error. Exit: operator/cleanup job purges staged object, or user retries (→ `PHOTO_PENDING`). |
| ARCHIVE_PENDING     | SYS-009 (Archive producer) | A new recipe version snapshot has been written to RDS and an archive job row exists but has not yet been enqueued/sent.   | Enter: version snapshot persisted. Exit: SQS enqueue acknowledged (→ `ARCHIVE_IN_FLIGHT`).                                                                            |
| ARCHIVE_IN_FLIGHT   | SYS-010 (Archive consumer) | Archive job has been delivered to the version-archive queue and the consumer is writing `versions/{recipe_id}/v{n}.json`. | Enter: SQS message in-flight to consumer. Exit: S3 PutObject acknowledged and pending row cleared (→ `NORMAL`); persistent failure routes to `ARCHIVE_DEAD_LETTER`.   |
| ARCHIVE_DEAD_LETTER | SYS-010 (Archive DLQ)      | Archive job exhausted retries and is parked on the DLQ awaiting operator triage/replay.                                   | Enter: max-receive exceeded on main queue. Exit: operator replays message back onto main queue (→ `ARCHIVE_IN_FLIGHT`) or marks the job permanently abandoned.        |
| ERASURE_REQUESTED   | SYS-013 (Erasure service)  | A GDPR erasure request has been accepted and a job ledger row created in `queued` state; no destructive work has begun.   | Enter: erasure API accepts the request. Exit: worker picks up the job (→ `ERASURE_IN_PROGRESS`).                                                                      |
| ERASURE_IN_PROGRESS | SYS-013 (Erasure worker)   | Erasure worker is purging owned RDS rows and S3 objects (photos + version archives) for the subject account.              | Enter: worker leases the job row. Exit: all targeted artifacts purged and ledger marked `completed` (→ `ERASURE_COMPLETED`); fatal failure leaves ledger in `failed`. |
| ERASURE_COMPLETED   | SYS-013 (Erasure ledger)   | Erasure job has reached a terminal `completed` state; only the audit/idempotency ledger row is retained for the subject.  | Enter: worker finishes purge successfully. Exit: terminal state. Re-requests are idempotent against this ledger row.                                                  |

### Design Notes

All components trace to explicit requirements listed in the Decomposition View. SYS-018 (Configuration and Environment Resolution) and SYS-019 (Observability and Alerting) are cross-cutting infrastructure components whose parent requirements are non-functional (REQ-NF-018/019/020 and REQ-041/REQ-NF-004/015/016/017 respectively). While their traceability to REQs is complete, their implementation pattern differs from the direct feature-requirement-to-component mapping of SYS-001 through SYS-017: they enforce constraints and provide operational infrastructure rather than implementing user-facing capabilities.

**Configuration ownership split (REQ-NF-018 / REQ-NF-019 / REQ-NF-020):** SYS-018 is the sole **provider** of typed configuration loading, default resolution, and environment-variable validation across API, web, and mobile. SYS-016 (Web Client) and SYS-017 (Mobile Client) are **consumers** of resolved configuration values and own correct runtime behavior given those values (e.g., honoring resolved API base URLs, port mappings, and feature flags). Downstream Matrix B verification SHOULD test loader/default behavior against SYS-018 and consumption/runtime behavior against SYS-016 (REQ-NF-018) and SYS-017 (REQ-NF-019); REQ-NF-020 is verified solely against SYS-018.

## Glossary

| Term               | Definition                                                                                                                                          |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tombstone          | Soft-delete state (`deleted_at` set) where a recipe is hidden from normal read paths but retained for compliance/history.                           |
| Pending archive    | A replayable row in `recipe_version_pending_archives` representing a version not yet confirmed in S3.                                               |
| Substantive edit   | An edit touching ingredients or instructions, used as privacy-unlock criterion for imported public-source clones.                                   |
| Snapshot clone     | Collection clone that copies source membership at clone time and does not auto-sync afterward.                                                      |
| Pull-from-source   | User-initiated reconcile action that updates cloned collection membership under explicit add/remove/no-overwrite rules.                             |
| Idempotent erasure | `POST /api/v1/account/erasure` behavior that reuses queued/running jobs, returns 410 for completed jobs, and creates fresh jobs only after failure. |
